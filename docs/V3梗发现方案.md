## 〇、给急着看结论的你（TL;DR）

V2 在 787 条评论上做到了「跨视频聚合 + 变体合并 + pairwise LLM judge」，但还有**三块真正的梗信号没有吃到**：

1. **弹幕刷屏共鸣**：同视频同时间点的密集相似弹幕 —— 目前 scout 根本没抓弹幕
2. **评论复读 + 点赞权重 + 带图**：相同/近似评论被多人复读、高赞评论（尤其带图）很可能是梗 —— 目前 like/image/复读检测都没接
3. **自学习闭环**：遇到不认识的候选 → 自动联网搜索 → 写回库 → 下次作为 few-shot 例子 —— 目前一次性跑完就结束，无累积

这三块正好对应**人类是怎么学习梗的**：弹幕刷屏给你一眼看出"大家都在说同一句"、高赞/复读评论让你注意到"这句被反复引用"、遇到不懂的就搜一下然后关联到场景。

V3 的工程目标就是把这三种人类直觉**翻译成工程信号**灌给 V2 现有的 KG + judge。不推翻 V2 的算法骨架，只补信号。

同时，**V3 直接融入生产管线**（不再以 `tools/embedding_cluster_mvp/pipeline_v3/` 的独立子树存在）：
- V3 的候选发现模块搬到 `meme_detector/candidate_discovery/`，替代 miner 的 per-bundle LLM 打分
- scout 扩展弹幕 + like + image，archivist 新增表，api / frontend 扩展视图
- researcher 现有的 web search + 词条化流程**保留**，输入从 miner bundle 改为接 `Community` + `BurstEvent` 证据
- V3 的自学习库**直接写 Meilisearch 的 meme index**（用 `source` 字段区分来源），不再单独建 `meme_library_entries`

实施拆 5 个 Milestone / 约 8 个 PR，最高 ROI 是 Milestone 1（弹幕）。

## 一、为什么做 V3

### 1.1 V2 实测到的瓶颈

v2_run_003（787 条评论，修复 co_occurs 污染后）：
- Meme 社区 26 个，但其中 **19 个是 size=1**（只被提了 1 次），没有聚到变体
- `华强买瓜` 和 `买瓜 / 买瓜宇宙` 被分成两个独立社区（embedding cos 未达 0.75 阈值）
- 34 个 `uncertain` 社区就此丢弃，没有后续补证机制
- 很多单视频下零散出现的 term 直接判成孤立社区 → `uncertain`

根因不在 V2 的算法，而在**输入信号太单薄**：
- 只有评论文本 + 候选 term 的 embedding
- 没有弹幕、没有 like_count、没有复读检测、没有图
- 没有历史梗库作为 few-shot

这导致 V2 的 LLM judge 在短语料场景下经常只有"一个孤立的 term"作为证据，它只能靠自己的先验知识判定，发挥不出"数据信号 + LLM 常识"的合力。

### 1.2 用户描述的人类学习梗的两条路径

> 第一种是从弹幕刷屏判断：视频里某些场景/内容出现时，大量弹幕刷一样或类似的吐槽来共鸣 → 这就是梗。不认识来源的话自己搜索学习，关联场景，下次再遇到就会用一样的梗。
>
> 第二种是从评论：高赞评论（尤其带图）往往是梗，但高赞也可能只是针对视频内容的独特幽默；**复读多的评论**是梗的概率更高。

拆开看，人类用了**五种信号**：
1. **时间窗共鸣**：同视频同场景密集刷一样的弹幕 → 模板化复用，强梗信号
2. **文本复读**：相同/近似评论被多人 ctrl-C/V → 梗的典型形态
3. **点赞权重**：高赞评论不等于梗，但是个强 prior
4. **带图**：高赞带图的评论里的图往往是梗（梗图）
5. **自学习闭环**：遇到不认识的 → 搜索 → 关联场景 → 内化进记忆，下次用

V2 目前只有"**跨视频共现**"这一个信号（还是弱版本）。

### 1.3 共同根因

V2 把**候选 term 作为主角**，把评论/视频/作者当成配角；但人类判定梗时，**"现象"本身**（弹幕墙、复读评论、梗图）才是主角。V3 的核心变化：把"**共鸣事件**"升级为一等公民节点，与候选 term 并列。

## 二、V3 目标与边界

### 2.1 必须达成

| # | 能力 | 可观测指标 |
|---|---|---|
| G1 | 弹幕"共鸣爆点"识别 | 同视频同 ±2s 窗口内 ≥ 5 条相似弹幕时产出 `BurstEvent` |
| G2 | 弹幕/评论共鸣事件进 KG | `BurstEvent` 节点存在、与关联 term 有 `triggered_by` 边 |
| G3 | 评论 like_count 加权 | social 打分 >= 0.8 的 term 里出现「高赞复读驱动」的新 meme |
| G4 | 评论近似复读检测 | 同视频下近似评论 ≥ 3 条产出 `RepeatCluster` |
| G5 | 自学习闭环最小可用版 | uncertain 社区自动触发 1 次 web search，结果写回 judge prompt；确认的 meme 写入 `meme_library` 供下轮 few-shot |
| G6 | V2 所有能力保留 | 原 V2 tests 全绿、原 v2_run_00x 行为不劣化 |

### 2.2 明确不做

- **不重写生产管线的基础设施层**（scout 抓取框架、archivist 存储、api、frontend、llm_factory、researcher 的 web search/词条化流程都保留；具体见第十节「淘汰/保留/扩展」三张表）
- 不做**梗的传播演化刻画**（同一梗的时间演化/热度衰减留待 V4）
- 不引入新 LLM 模型（continue with existing providers）
- 不做**图片的视觉相似聚类**（第一版只做 OCR/caption 后的文本 embedding，视觉 emb 留后续）
- 不动 `meme_records` 主表的 schema 破坏性改动（只加 `source` 扩枚举 + `discovery_community_id` 可空外键；不破坏现有数据）

### 2.3 成功判据

在 5k-10k 条评论 + 同等数量弹幕的输入上：

| 指标 | 目标 |
|---|---|
| meme recall（相对 V2 同样本） | 相对提升 ≥ 30% |
| meme precision | 不劣化（≥ V2 水平） |
| uncertain 社区经过自学习后被明确分类的比例 | ≥ 50% |
| 端到端耗时 | ≤ V2 × 2 |
| 每 1k 评论额外成本（相对 V2） | ≤ ¥5 |

任意一项严重不达标 → 回滚对应 Milestone 的改动。

## 三、核心 pipeline 变化

### 3.1 V2 现有 pipeline（基线）

```
拉评论 → Layer1 LLM 抽取 → 聚合候选 → embedding → 构建 KG
→ variant_only_subgraph → Leiden → compute_communities → pairwise LLM judge → 输出
```

### 3.2 生产管线融合后的 pipeline

**旧生产管线**：
```
scout → miner(LLM per-bundle 打分) → researcher(LLM 深挖 + 词条化) → meili
```

**V3 融入后的生产管线**：
```
scout(+弹幕 / +like_count / +image_urls)                 ← M1-a 扩展
    ↓
pre-cluster（事件检测）                                   ← M1-b / M2-a 新增
  - 弹幕时间窗共鸣 → BurstEvent
  - 评论近似复读     → RepeatCluster
    ↓
candidate-discovery（替代 miner 的 LLM 打分）              ← M1-c / M2-b 新增
  Layer1 LLM 抽取 → 聚合候选 → embedding → 构建 KG
    (candidate / comment / danmaku / video / author
     + BurstEvent / RepeatCluster)
  → variant_only_subgraph → Leiden
  → compute_communities（注入 burst / repeat / likes / library 统计）
    ↓
pre-judge 自学习补证                                       ← M3 新增
  - 命中 Meili 主 meme index → strong meme prior
  - uncertain 倾向 → 1 次 web search（复用 researcher.tools）
    ↓
pairwise LLM judge（输出 Community + verdict）
    ↓
researcher（保留，输入改为 Community + BurstEvent 证据）   ← M4 改造
  - 对 verdict=meme 的 Community 做 web search 深挖 + 词条化
  - uncertain 社区进 Workbench 等人工裁决
    ↓
meili 主 meme index（source='v3_auto' / 'manual' / 'researcher'）
```

**关键变化**：miner 的 `scorer.py`（per-bundle LLM 打分）被整个替换为 candidate-discovery 层。miner 的 `bundler.py` 和 `video_context.py`（证据聚合工具）**保留**，但服务对象从"为 LLM 打分准备输入"变成"为社区提供原始证据"。

**关键设计决策**：

| 决策 | 理由 |
|---|---|
| BurstEvent / RepeatCluster 作为 KG 节点，不是 term 上的 attribute | 一个 BurstEvent 可能触发多个 term；用节点表达更自然，也方便 judge prompt 引用 |
| Leiden 输入仍是 variant_only_subgraph | 新事件节点不进 Leiden 的连通性（避免回归 co_occurs 污染）；只在 compute_communities 阶段注入统计 |
| 自学习库复用 Meilisearch 主 meme index | 生产管线已有主 meme index（`archivist/meili_store`）；V3 的 `source='v3_auto'` 写入，researcher 可后续 promote 为 `source='researcher'` |
| 淘汰 miner/scorer.py，保留 miner/bundler.py + video_context.py | `scorer.py` 是 per-bundle 的 LLM 打分 MVP，已被 V3 的 KG+judge 替代；`bundler.py`/`video_context.py` 是纯证据聚合工具，新旧流程都依赖它们 |
| researcher 保留 decider + taxonomy + tools | 这些是「从候选 → 词条化 meme」的业务逻辑，V3 的候选发现层把输入从 bundle 换成 Community 即可，不重写 |

### 4.2 Pre-cluster 事件检测

#### 4.2.1 弹幕共鸣爆点（BurstEvent）

**定义**：同一 `bvid` 下，**时间窗 ±W 秒**内出现 **≥ K 条近似弹幕** → 一个 `BurstEvent`。

**算法**：

```python
# meme_detector/candidate_discovery/burst_detector.py 伪代码

@dataclass
class BurstEvent:
    event_id: str           # f"burst::{bvid}::{center_ts:.1f}::{signature}"
    bvid: str
    center_ts: float        # 时间窗中心（秒）
    window_sec: float       # W（默认 3.0）
    signature: str          # 代表性弹幕文本（最长公共子串 or 最高频）
    danmaku_count: int      # 窗内相似弹幕数
    unique_users: int       # 去重用户数
    member_danmaku_ids: list[str]


def detect_burst_events(
    danmaku: list[dict],
    *,
    window_sec: float = 3.0,
    min_count: int = 5,
    min_unique_users: int = 3,
    text_sim_threshold: float = 0.75,
    embed_fn: Callable = embedding_for_short_text,
) -> list[BurstEvent]:
    """
    1) 按 bvid 分组
    2) 每组按 timestamp 排序
    3) 滑动窗口扫描，窗口内任意两条弹幕 embedding cos ≥ threshold → 算作"相似"
    4) 在窗口内找 connected components（近似弹幕簇）
    5) 簇大小 ≥ min_count 且 unique_users ≥ min_unique_users → BurstEvent
    6) 簇内取最高频或最长公共子串作为 signature
    """
```

**参数经验值**（可调）：
- W = 3.0 秒：B 站弹幕发送延迟容差
- K = 5：小于 5 条容易是巧合
- unique_users ≥ 3：防止单人刷屏
- sim threshold = 0.75：和 V2 variant 阈值对齐

**复杂度**：每个 bvid 的弹幕两两对比 embedding，N = 视频弹幕数。对于 5k 评论量级的视频集合（每个视频 100-500 弹幕），embedding 总量可控；真正的数据瓶颈在 M1 的抓取。

**输出**：`BurstEvent` 列表，序列化到 `burst_events.jsonl`。

**Milestone**：M1 后段

#### 4.2.2 评论近似复读（RepeatCluster）

**定义**：同一 `bvid` 下，**≥ N 条文本高度相似的评论** → 一个 `RepeatCluster`。

跨视频复读不在 v3.0 范围（对应场景是"同一梗同一字面被搬到多个视频"；V2 的跨视频 variant 边已经部分覆盖了）。

**算法**：

```python
# meme_detector/candidate_discovery/repeat_detector.py 伪代码

@dataclass
class RepeatCluster:
    cluster_id: str          # f"repeat::{bvid}::{idx}"
    bvid: str
    signature: str           # 代表评论文本
    member_count: int
    unique_authors: int
    weighted_likes: float    # Σ log(1 + like_count)
    member_comment_ids: list[str]


def detect_repeat_clusters(
    comments: list[dict],
    *,
    min_count: int = 3,
    text_sim_threshold: float = 0.85,  # 评论阈值比弹幕严
) -> list[RepeatCluster]:
    """
    1) 按 bvid 分组
    2) 评论 embedding（整条评论，不是 term）
    3) MinHash / 近邻搜索 → 找相似度 ≥ threshold 的簇
    4) 簇大小 ≥ min_count 且 unique_authors ≥ 2 → RepeatCluster
    """
```

**注意**：如果评论长度差异大（例如「绷不住」vs「绷不住了，这up太会玩了」），建议在 embedding 前做**句法首部对齐**（取前 N 字），否则长评论把短的稀释。

**Milestone**：M2

### 4.3 KG 扩展

V2 现有节点：`candidate / comment / video / author`
V2 现有边：`contains / posted_on / posted_by / co_occurs / variant / variant+co_occurs`

**V3 新增节点类型**：

| 类型 | attrs | 说明 |
|---|---|---|
| `danmaku` | `text, timestamp, send_time, user_hash_id` | 弹幕节点，新粒度；只有进入 BurstEvent 的才建节点（避免全量弹幕爆炸） |
| `burst_event` | `center_ts, signature, danmaku_count, unique_users, weighted_count` | 共鸣爆点 |
| `repeat_cluster` | `signature, member_count, unique_authors, weighted_likes` | 评论复读簇 |
| `meme_library_entry` | `entry_id, title, aliases, source, last_seen_at` | 已确认梗库条目（自学习闭环产出） |

**V3 新增边类型**：

| 边 | 头 → 尾 | 说明 |
|---|---|---|
| `burst_member` | `danmaku → burst_event` | 弹幕属于某 burst |
| `burst_occurs_on` | `burst_event → video` | 爆点发生在某视频 |
| `burst_triggers` | `burst_event → candidate` | 爆点内弹幕文本命中候选 term |
| `repeat_member` | `comment → repeat_cluster` | 评论属于某复读簇 |
| `repeat_contains` | `repeat_cluster → candidate` | 复读簇的 signature 命中候选 term |
| `matches_library` | `candidate → meme_library_entry` | 候选命中已知梗 |

**Leiden 输入不变**：仍用 `variant_only_subgraph`（只含 candidate 节点 + variant 边），新事件节点**不参与社区划分**。

**compute_communities 注入新统计**：

### 4.4 自学习闭环（Milestone 3）

#### 4.4.1 meme library 持久化 —— 直接写 Meilisearch 主 index

**v0.2 变更**：不再单独建 `meme_library_entries` 表，**直接复用 `meme_records` 主表 + Meilisearch 主 meme index**。区分来源用既有/新增的 `source` 字段：

- 原有 researcher 深挖产出 → `source = 'researcher'`（或 `'manual'`）
- V3 candidate-discovery + judge 产出 → `source = 'v3_auto'`（新增枚举值）
- Researcher 对 `v3_auto` 词条 promote 到完整词条 → `source` 改写为 `'researcher'`

这意味着：
- `meme_records` 表新增/确认 `source` 字段 + `discovery_community_id` 外键（可空）
- `archivist/meili_store.py` 需要 index 新的 source facet，便于 researcher / 前端按来源筛选
- 冷启动时所有 V3 产出都是 `source='v3_auto'`，researcher 消费这些 community 做深挖，promote 成 `'researcher'`

V3 每次 run 结束，对 `verdict=meme` 的社区：
- 查 Meili 主 index 模糊命中 → append aliases、刷新 `last_seen_at` + EMA 置信度
- 否则 → 写入新 record（`source='v3_auto'`，附 evidence JSON：`community_id / burst_events / repeat_clusters / top_comments`）

**researcher 消费路径**（M4）：api 新增 `/api/researcher/queue` 按 `source='v3_auto'` 出队，researcher agent 执行深挖后写回 `source='researcher'`。

#### 4.4.2 Pre-judge 库匹配 + web search

```python
# meme_detector/candidate_discovery/learner.py 伪代码

async def enrich_community_before_judge(
    community: Community,
    meili_meme_index: MeiliMemeIndex,     # 直接复用 archivist/meili_store
    web_search_fn: Callable,              # 直接 import researcher.tools.volcengine_web_search_summary
) -> CommunityEnrichment:
    """
    1) 在 meili_meme_index 里查社区 term 的精确/模糊匹配
       - 精确命中 → strong_prior=meme（绕过 web search）
       - 模糊命中（aliases 或 embedding） → weak_prior=meme + 引用理由
    2) 若无库命中且社区元信息显示可能是 uncertain：
       - 取社区 top1 term 做 1 次 web search
       - 把 search summary 作为 judge prompt 的额外上下文
    3) 频控：每次 run 最多 N 次 web search（默认 **10**）。8 以内仍没返回有用 summary 就当 `uncertain` 处理（用户 Q4：10 次都得不到结果 ≈ 不存在）。
    """
```

**费控上限**：每次 run web search 总数 ≤ **10**（Q4 用户确认），跑完 87 条评论也只花 ~¥0.5。

**Milestone**：M3

### 5.2 生产管线代码的处置

见第十节「淘汰/保留/扩展」三张表。核心原则：
- **决策层**（per-bundle LLM 打分）→ 淘汰
- **基础设施层**（scout 抓取、archivist 存储、api、frontend、llm_factory、miner/bundler、researcher 工具） → 保留

| 阶段 | 生产 `/api/pipeline/run` 行为 | rollback 方式 |
|---|---|---|
| Phase 0（现状） | scout → miner/scorer → researcher → meili | — |
| Phase 1（M1-a 合入后） | 同 Phase 0；scout 写入新字段但旧逻辑不读 | 零影响 |
| Phase 2（M1-c 合入后） | scout → **candidate_discovery** → researcher(改输入) → meili；`miner/scorer.py` 从此不在生产路径调用 | `git revert` M1-c |
| Phase 3（M2/M3 合入后） | 新路径逐步叠 repeat / like / image / learner 信号 | `git revert` 对应 PR |
| Phase 4（M4-b 合入后，纯清理） | 删除 `miner/scorer.py` 源文件、删除 legacy import、清理死代码 | 需要重新 commit scorer 源码（不做此步前 Phase 2-3 阶段 scorer 仍在仓库内，可随时 revert 回用） |

关键含义：
- **M1-c 合入当天 = 生产切新路径当天**。合入前需要跑完一批用旧 scorer 的任务（若有未跑完的），避免中途切。
- Phase 2-3 期间，`miner/scorer.py` **仍在仓库里但不被生产调用**。万一 candidate-discovery 路径崩了，最快的 rollback 是在 `pipeline_service.py` 里改一行回到老分支，不用 revert 新代码。
- Phase 4（M4-b）做纯清理后，scorer 源文件删除；再之后的 rollback 必须 revert M4-b。

## 七、风险与备选

| # | 风险 | 概率 | 备选方案 |
|---|---|---|---|
| R1 | B 站弹幕接口限流 / 需要登录 cookie | 中 | bilibili-api 已有会话认证支持；实测失败则降级为"仅评论"模式，Milestone 1 部分降档 |

| R7 | KG 节点膨胀拖慢 Leiden | 低 | 新节点不进 variant_only_subgraph，Leiden 输入规模不变 |
| R8 | V2 既有 test 受影响 | 低 | V2 代码不改，V3 走独立模块；CI 应绿 |

## 十二、一页纸总结

- **V2 做了什么**：跨视频候选聚合 + 变体合并 + LLM judge（干净、够用）
- **V2 缺什么**：弹幕信号、评论复读、点赞权重、带图、自学习库
- **V3 做什么**：把人类判定梗的 5 类信号灌给 V2 现有 KG + judge
- **V3 怎么做**：加 `BurstEvent / RepeatCluster` 两类事件节点 + 直接写 Meili 主 meme index 的自学习闭环
- **V3 和生产管线的关系**：**融入，不并存**。candidate-discovery 作为生产管线里 miner/scorer 的替代层，researcher 输入从 bundle 改为 Community；scout/archivist/api/frontend/llm_factory/miner.bundler/researcher.tools 全部保留
- **淘汰**：`miner/scorer.py`（per-bundle LLM 打分） / pipeline_v2 在生产的入口 / V3 独立梗库表
- **Leiden 输入仍是 variant_only_subgraph**（避免 co_occurs 污染）
- **预算**：比当前生产多 ~¥2/1k 评论、多 ~25% 耗时
- **分 5 个 Milestone / 约 8 个 PR**，最高 ROI 是 M1（弹幕 + candidate-discovery 骨架）
- **上线节奏**：**无 env flag**。M1-c 合入即在生产路径替换 `miner/scorer.py`（scorer 源文件仍在仓库便于 rollback）；M2/M3 逐步叠信号；M4-b 纯清理删 scorer 源文件。
