## 〇、给急着看结论的你（TL;DR）

V2 在 787 条评论上做到了「跨视频聚合 + 变体合并 + pairwise LLM judge」，但还有**三块真正的梗信号没有吃到**：

1. **弹幕刷屏共鸣**：同视频同时间点的密集相似弹幕 —— 目前 scout 根本没抓弹幕
2. **评论复读 + 点赞权重 + 带图**：相同/近似评论被多人复读、高赞评论（尤其带图）很可能是梗 —— 目前 like/image/复读检测都没接
3. **自学习闭环**：遇到不认识的候选 → 自动联网搜索 → 写回库 → 下次作为 few-shot 例子 —— 目前一次性跑完就结束，无累积

这三块正好对应**人类是怎么学习梗的**：弹幕刷屏给你一眼看出"大家都在说同一句"、高赞/复读评论让你注意到"这句被反复引用"、遇到不懂的就搜一下然后关联到场景。

V3 的工程目标就是把这三种人类直觉**翻译成工程信号**灌给 V2 现有的 KG + judge。不推翻 V2，只补信号。

实施拆 4 个 Milestone / 6-8 个 PR，最高 ROI 是 Milestone 1（弹幕）。

## 一、为什么做 V3

### 1.1 V2 实测到的瓶颈

v2_run_003（787 条评论，修复 co_occurs 污染后）：
- Meme 社区 26 个，但其中 **19 个是 size=1**（只被提了 1 次），没有聚到变体
- `华强买瓜` 和 `买瓜 / 买瓜宇宙` 被分成两个独立社区（embedding cos 未达 0.75 阈值）
- 34 个 `uncertain` 社区就此丢弃，没有后续补证机制
- 很多单视频下零散出现的 term 直接判成孤立社区 → `uncertain`

根因不在 V2 的算法，而在**输入信号太单薄**：
- 只有评论文本 + 候选 term 的 embedding
- 没有弹幕、没有 like_count、没有复读检测、没有图
- 没有历史梗库作为 few-shot

这导致 V2 的 LLM judge 在短语料场景下经常只有"一个孤立的 term"作为证据，它只能靠自己的先验知识判定，发挥不出"数据信号 + LLM 常识"的合力。

### 1.2 用户描述的人类学习梗的两条路径

> 第一种是从弹幕刷屏判断：视频里某些场景/内容出现时，大量弹幕刷一样或类似的吐槽来共鸣 → 这就是梗。不认识来源的话自己搜索学习，关联场景，下次再遇到就会用一样的梗。
>
> 第二种是从评论：高赞评论（尤其带图）往往是梗，但高赞也可能只是针对视频内容的独特幽默；**复读多的评论**是梗的概率更高。

拆开看，人类用了**五种信号**：
1. **时间窗共鸣**：同视频同场景密集刷一样的弹幕 → 模板化复用，强梗信号
2. **文本复读**：相同/近似评论被多人 ctrl-C/V → 梗的典型形态
3. **点赞权重**：高赞评论不等于梗，但是个强 prior
4. **带图**：高赞带图的评论里的图往往是梗（梗图）
5. **自学习闭环**：遇到不认识的 → 搜索 → 关联场景 → 内化进记忆，下次用

V2 目前只有"**跨视频共现**"这一个信号（还是弱版本）。

### 1.3 共同根因

V2 把**候选 term 作为主角**，把评论/视频/作者当成配角；但人类判定梗时，**"现象"本身**（弹幕墙、复读评论、梗图）才是主角。V3 的核心变化：把"**共鸣事件**"升级为一等公民节点，与候选 term 并列。

## 二、V3 目标与边界

### 2.1 必须达成

| # | 能力 | 可观测指标 |
|---|---|---|
| G1 | 弹幕"共鸣爆点"识别 | 同视频同 ±2s 窗口内 ≥ 5 条相似弹幕时产出 `BurstEvent` |
| G2 | 弹幕/评论共鸣事件进 KG | `BurstEvent` 节点存在、与关联 term 有 `triggered_by` 边 |
| G3 | 评论 like_count 加权 | social 打分 >= 0.8 的 term 里出现「高赞复读驱动」的新 meme |
| G4 | 评论近似复读检测 | 同视频下近似评论 ≥ 3 条产出 `RepeatCluster` |
| G5 | 自学习闭环最小可用版 | uncertain 社区自动触发 1 次 web search，结果写回 judge prompt；确认的 meme 写入 `meme_library` 供下轮 few-shot |
| G6 | V2 所有能力保留 | 原 V2 tests 全绿、原 v2_run_00x 行为不劣化 |

### 2.2 明确不做

- 不动生产管线的 scout/miner/researcher 现有流程（只复用其子工具，例如 web search 函数、Meilisearch 库）
- 不做**梗库的用户态 UI**（新增 meme 库只提供读写 API，不做前端改造）
- 不做**梗的传播演化刻画**（同一梗的时间演化/热度衰减留待 V4）
- 不引入新 LLM 模型（continue with existing providers）
- 不做**图片的视觉相似聚类**（第一版只做 OCR/caption 后的文本 embedding，视觉 emb 留后续）

### 2.3 成功判据

在 5k-10k 条评论 + 同等数量弹幕的输入上：

| 指标 | 目标 |
|---|---|
| meme recall（相对 V2 同样本） | 相对提升 ≥ 30% |
| meme precision | 不劣化（≥ V2 水平） |
| uncertain 社区经过自学习后被明确分类的比例 | ≥ 50% |
| 端到端耗时 | ≤ V2 × 2 |
| 每 1k 评论额外成本（相对 V2） | ≤ ¥5 |

任意一项严重不达标 → 回滚对应 Milestone 的改动。

## 三、核心 pipeline 变化

### 3.1 V2 现有 pipeline（基线）

```
拉评论 → Layer1 LLM 抽取 → 聚合候选 → embedding → 构建 KG
→ variant_only_subgraph → Leiden → compute_communities → pairwise LLM judge → 输出
```

### 3.2 V3 pipeline（增量）

```
Scout 扩展：                                         ← 新增 M1
  拉评论 + 拉弹幕 + 抓图片 URL + like_count
                                                         ↓
Layer1 抽取：
  评论走原路径 + 弹幕走简化抽取（时间点聚合）
                                                         ↓
Pre-cluster 事件检测：                                 ← 新增 M1/M2
  - 弹幕同窗共鸣 → BurstEvent
  - 评论近似复读 → RepeatCluster
                                                         ↓
构建 KG（多一类节点 + 多种边）：
  candidate / comment / danmaku / video / author
  + BurstEvent / RepeatCluster / MemeLibraryEntry     ← 新
                                                         ↓
variant_only_subgraph → Leiden                        ← 不变
                                                         ↓
compute_communities（社区元信息扩展）：                 ← 扩展
  + burst_event_count（命中共鸣爆点数）
  + repeat_cluster_count（命中复读簇数）
  + total_weighted_likes（社区内评论的 Σ log(1+likes)）
  + matched_library_entries（命中已有梗库的条数）
                                                         ↓
Pre-judge 自学习补证：                                 ← 新增 M3
  - 若社区命中 meme_library：打 strong meme prior
  - 否则：对 uncertain 倾向的社区触发 1 次 web search
                                                         ↓
pairwise LLM judge（prompt 吃更多信号）：              ← prompt 扩展
  + burst/repeat 辅助信号
  + 已知梗库匹配
  + web search 结果
                                                         ↓
后处理：                                               ← 新增 M3
  - meme verdict → 写入 meme_library 供下轮
  - 统计 metrics → eval_report_v3.md
```

**关键设计决策**：

| 决策 | 理由 |
|---|---|
| BurstEvent / RepeatCluster 作为 KG 节点，不是 term 上的 attribute | 一个 BurstEvent 可能触发多个 term；用节点表达更自然，也方便 judge prompt 引用 |
| Leiden 输入仍是 variant_only_subgraph | 新事件节点不进 Leiden 的连通性（避免回归 co_occurs 污染）；只在 compute_communities 阶段注入统计 |
| 自学习闭环的库复用生产管线的 Meilisearch | 生产管线已有 meme 词条库（agent_conversations 不复用，只读 meme records）；避免二次建表 |

### 4.2 Pre-cluster 事件检测

#### 4.2.1 弹幕共鸣爆点（BurstEvent）

**定义**：同一 `bvid` 下，**时间窗 ±W 秒**内出现 **≥ K 条近似弹幕** → 一个 `BurstEvent`。

**算法**：

```python
# tools/embedding_cluster_mvp/pipeline_v3/burst_detector.py 伪代码

@dataclass
class BurstEvent:
    event_id: str           # f"burst::{bvid}::{center_ts:.1f}::{signature}"
    bvid: str
    center_ts: float        # 时间窗中心（秒）
    window_sec: float       # W（默认 3.0）
    signature: str          # 代表性弹幕文本（最长公共子串 or 最高频）
    danmaku_count: int      # 窗内相似弹幕数
    unique_users: int       # 去重用户数
    member_danmaku_ids: list[str]


def detect_burst_events(
    danmaku: list[dict],
    *,
    window_sec: float = 3.0,
    min_count: int = 5,
    min_unique_users: int = 3,
    text_sim_threshold: float = 0.75,
    embed_fn: Callable = embedding_for_short_text,
) -> list[BurstEvent]:
    """
    1) 按 bvid 分组
    2) 每组按 timestamp 排序
    3) 滑动窗口扫描，窗口内任意两条弹幕 embedding cos ≥ threshold → 算作"相似"
    4) 在窗口内找 connected components（近似弹幕簇）
    5) 簇大小 ≥ min_count 且 unique_users ≥ min_unique_users → BurstEvent
    6) 簇内取最高频或最长公共子串作为 signature
    """
```

**参数经验值**（可调）：
- W = 3.0 秒：B 站弹幕发送延迟容差
- K = 5：小于 5 条容易是巧合
- unique_users ≥ 3：防止单人刷屏
- sim threshold = 0.75：和 V2 variant 阈值对齐

**复杂度**：每个 bvid 的弹幕两两对比 embedding，N = 视频弹幕数。对于 5k 评论量级的视频集合（每个视频 100-500 弹幕），embedding 总量可控；真正的数据瓶颈在 M1 的抓取。

**输出**：`BurstEvent` 列表，序列化到 `burst_events.jsonl`。

**Milestone**：M1 后段

#### 4.2.2 评论近似复读（RepeatCluster）

**定义**：同一 `bvid` 下，**≥ N 条文本高度相似的评论** → 一个 `RepeatCluster`。

跨视频复读不在 v3.0 范围（对应场景是"同一梗同一字面被搬到多个视频"；V2 的跨视频 variant 边已经部分覆盖了）。

**算法**：

```python
# tools/embedding_cluster_mvp/pipeline_v3/repeat_detector.py 伪代码

@dataclass
class RepeatCluster:
    cluster_id: str          # f"repeat::{bvid}::{idx}"
    bvid: str
    signature: str           # 代表评论文本
    member_count: int
    unique_authors: int
    weighted_likes: float    # Σ log(1 + like_count)
    member_comment_ids: list[str]


def detect_repeat_clusters(
    comments: list[dict],
    *,
    min_count: int = 3,
    text_sim_threshold: float = 0.85,  # 评论阈值比弹幕严
) -> list[RepeatCluster]:
    """
    1) 按 bvid 分组
    2) 评论 embedding（整条评论，不是 term）
    3) MinHash / 近邻搜索 → 找相似度 ≥ threshold 的簇
    4) 簇大小 ≥ min_count 且 unique_authors ≥ 2 → RepeatCluster
    """
```

**注意**：如果评论长度差异大（例如「绷不住」vs「绷不住了，这up太会玩了」），建议在 embedding 前做**句法首部对齐**（取前 N 字），否则长评论把短的稀释。

**Milestone**：M2

### 4.3 KG 扩展

V2 现有节点：`candidate / comment / video / author`
V2 现有边：`contains / posted_on / posted_by / co_occurs / variant / variant+co_occurs`

**V3 新增节点类型**：

| 类型 | attrs | 说明 |
|---|---|---|
| `danmaku` | `text, timestamp, send_time, user_hash_id` | 弹幕节点，新粒度；只有进入 BurstEvent 的才建节点（避免全量弹幕爆炸） |
| `burst_event` | `center_ts, signature, danmaku_count, unique_users, weighted_count` | 共鸣爆点 |
| `repeat_cluster` | `signature, member_count, unique_authors, weighted_likes` | 评论复读簇 |
| `meme_library_entry` | `entry_id, title, aliases, source, last_seen_at` | 已确认梗库条目（自学习闭环产出） |

**V3 新增边类型**：

| 边 | 头 → 尾 | 说明 |
|---|---|---|
| `burst_member` | `danmaku → burst_event` | 弹幕属于某 burst |
| `burst_occurs_on` | `burst_event → video` | 爆点发生在某视频 |
| `burst_triggers` | `burst_event → candidate` | 爆点内弹幕文本命中候选 term |
| `repeat_member` | `comment → repeat_cluster` | 评论属于某复读簇 |
| `repeat_contains` | `repeat_cluster → candidate` | 复读簇的 signature 命中候选 term |
| `matches_library` | `candidate → meme_library_entry` | 候选命中已知梗 |

**Leiden 输入不变**：仍用 `variant_only_subgraph`（只含 candidate 节点 + variant 边），新事件节点**不参与社区划分**。

**compute_communities 注入新统计**：

```python
@dataclass
class Community:
    # V2 现有字段
    community_id: int
    size: int
    total_freq: int
    n_videos: int
    n_authors: int
    n_comments: int
    internal_density: float
    avg_variant_sim: float
    burst_score: float            # V2 里叫 burst_score（跨时间爆发），语义上与 V3 BurstEvent 不冲突
    cross_video_ratio: float
    cooccur_edge_count: int
    cooccur_event_count: int

    # V3 新增
    burst_event_count: int        # 社区内候选命中 BurstEvent 的次数
    burst_weighted_danmaku: int   # 命中 BurstEvent 的弹幕总数（加权强度）
    repeat_cluster_count: int     # 社区内候选命中 RepeatCluster 的次数
    total_weighted_likes: float   # 社区内 comment 的 Σ log(1 + likes)
    matched_library_titles: list[str]  # 命中已知梗库的 title 列表
```

**Milestone**：M1 建 BurstEvent 相关、M2 建 RepeatCluster 相关、M3 建 MemeLibraryEntry 相关

### 4.4 自学习闭环（Milestone 3）

#### 4.4.1 meme_library 持久化

**复用生产管线的 Meilisearch + DuckDB**。新增一张 DuckDB 表：

```sql
CREATE TABLE IF NOT EXISTS meme_library_entries (
    entry_id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    aliases JSON NOT NULL DEFAULT '[]',
    source TEXT NOT NULL,  -- 'v3_auto' | 'manual' | 'v3_rewrite'
    first_seen_at TIMESTAMP NOT NULL,
    last_seen_at TIMESTAMP NOT NULL,
    confidence REAL NOT NULL,
    evidence JSON NOT NULL DEFAULT '{}'
);
```

V3 每次 run 结束，对 `verdict=meme` 的社区：
- 若 title 命中已有 entry → 追加 aliases、更新 last_seen_at + confidence（EMA）
- 否则 → 插入新 entry

**不覆盖生产管线的 `meme_records` 表**（生产管线有独立的 researcher 产出流程）。V3 的库是**只读给 judge 吃 few-shot**，写入方向是单向的。

#### 4.4.2 Pre-judge 库匹配 + web search

```python
# tools/embedding_cluster_mvp/pipeline_v3/learner.py 伪代码

async def enrich_community_before_judge(
    community: Community,
    library: MemeLibrary,
    web_search_fn: Callable,
) -> CommunityEnrichment:
    """
    1) 在库里查社区 term 的精确/模糊匹配
       - 精确命中 → strong_prior=meme（绕过 web search）
       - 模糊命中（aliases 或 embedding） → weak_prior=meme + 引用理由
    2) 若无库命中且社区元信息显示可能是 uncertain：
       - 取社区 top1 term 做 1 次 web search（复用 researcher.tools.volcengine_web_search_summary）
       - 把 search summary 作为 judge prompt 的额外上下文
    3) 频控：每次 run 最多 N 次 web search（默认 20）
    """
```

**费控上限**：每次 run web search 总数 ≤ 20，跑完 87 条评论也只花 ~¥1。

**Milestone**：M3

#### 4.4.3 judge prompt 扩展

V2 judge prompt 的 stats 块：
```
出现 X 次 · 跨 Y 视频 · 跨 Z 作者 · 簇内相似度均值 S · 跨视频率 R · 簇内共现事件 E 次 / 共现边 B 条
```

V3 扩展为：
```
出现 X 次 · 跨 Y 视频 · 跨 Z 作者 · 簇内相似度均值 S · 跨视频率 R · 簇内共现事件 E 次
弹幕共鸣爆点：命中 P 次（窗内弹幕 D 条、代表文本「XXX」）     ← 新
评论复读簇：命中 Q 次（代表文本「XXX」，去重作者 U）         ← 新
社区加权点赞：Σ log(1+likes) = L.LL                            ← 新
已知梗库匹配：[title1, title2]（来源：v3_auto）                ← 新（若命中）
Web 检索补证：前三条摘要 …                                     ← 新（若触发）
```

**Milestone**：M3 prompt 改、M1/M2 字段就绪

## 五、与 V2 的关系

| 维度 | V3 做法 |
|---|---|
| V2 代码 | **不动**。V3 通过 import 复用 V2 的 `graph_builder / community / judge / extractor / embedder` |
| V2 行为 | 不劣化。V2 的 tests 不改，V2 的 run.py 不改，可以继续按原方式跑 |
| V2 的 KG | V3 的 KG = V2 的 KG + 新节点/新边。只要 V3 graph_builder 把新边 **打上 rel 标签**，variant_only_subgraph 默认会剔除，Leiden 不受影响 |
| 切换方式 | 用户可选择跑 `pipeline_v2.run` 或 `pipeline_v3.run`，两个入口并存 |

**升级不是替换**。V2 稳定后，V3 作为可选路径，等 v3_run_00x 在足够样本上验证收益后，才考虑把 v3 设成默认。

## 六、成本与性能预估

### 6.1 额外成本（相对 V2，按 1k 评论 / 5k 弹幕估算）

| 环节 | 增量成本 |
|---|---|
| 弹幕抓取（B 站 API） | 免费 |
| 弹幕 embedding（5k × 1024 维） | DashScope ~¥0.5 |
| 评论 embedding 复读检测（1k × 1024 维） | ~¥0.1 |
| Web search 补证（20 次上限） | volcengine ~¥1 |
| Meilisearch 读写 | 本地 |
| **合计** | **~¥2/1k 评论** |

V2 是 ~¥1-2/1k 评论，V3 是 V2 + 约 ¥2 → **仍然远低于目标上限 ¥5**。

### 6.2 耗时

| 环节 | V2 基线（787 条） | V3 预估（同样本） |
|---|---|---|
| Layer1 抽取 | ~20 min | ~20 min（不变） |
| 弹幕抓取 + burst 检测 | - | ~5 min |
| 评论复读检测 | - | ~2 min |
| 图构建 + Leiden | < 1 min | ~2 min |
| pre-judge 补证（20 次 web search） | - | ~3 min |
| judge | ~15 min | ~15 min |
| **总** | ~40 min | **~50 min** |

**V3 预估 ≤ V2 × 1.3**（目标是 V2 × 2 内）。

## 七、风险与备选

| # | 风险 | 概率 | 备选方案 |
|---|---|---|---|
| R1 | B 站弹幕接口限流 / 需要登录 cookie | 中 | bilibili-api 已有会话认证支持；实测失败则降级为"仅评论"模式，Milestone 1 部分降档 |
| R2 | 弹幕文本短、embedding 质量差 | 中 | 可叠加字符 n-gram Jaccard 作为辅助距离，或退回字面精确/包含匹配 |
| R3 | RepeatCluster 在短语料上触发不足（同 bvid 评论 < 3 条） | 中 | M2 改为 `min_count=2`，同时把长度过短（< 3 字）的排除；在更大样本上再调回 3 |
| R4 | meme_library 在冷启动时没有条目 → 自学习收益为 0 | 高 | M3 先从 uncertain → web search → 回写开始，累积几轮后再谈 "命中库 → strong prior" |
| R5 | Web search 结果噪声（搜到不相关词条） | 中 | prompt 里明确标注 "仅作参考、不能自动提升为 meme"；judge LLM 自行权衡 |
| R6 | BurstEvent 把「大家都在刷666」识别成梗 | 中 | judge 里的 `old_general` 类天然会把 666 归入，不改架构；必要时在 Layer1 就把 `666` 打为 `platform_term`/`generic_phrase` |
| R7 | KG 节点膨胀拖慢 Leiden | 低 | 新节点不进 variant_only_subgraph，Leiden 输入规模不变 |
| R8 | V2 既有 test 受影响 | 低 | V2 代码不改，V3 走独立模块；CI 应绿 |

## 十、与生产管线的关系

V3 仍然**不进生产代码路径**。但复用范围比 V2 广：

| V3 复用生产的部分 | 复用方式 |

## 十二、一页纸总结

- **V2 做了什么**：跨视频候选聚合 + 变体合并 + LLM judge（干净、够用）
- **V2 缺什么**：弹幕信号、评论复读、点赞权重、带图、自学习库
- **V3 做什么**：把人类判定梗的 5 类信号灌给 V2 现有 KG + judge
- **V3 怎么做**：加 `BurstEvent / RepeatCluster / MemeLibraryEntry` 三类事件节点 + web search 自学习闭环
- **V3 不动什么**：V2 代码不改；生产管线不改；Leiden 输入仍是 variant_only_subgraph
- **预算**：比 V2 多 ~¥2/1k 评论、多 ~25% 耗时
- **分 4 个 Milestone、6-8 个 PR**，最高 ROI 是 M1（弹幕）
