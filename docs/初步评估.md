● MemeDetector 方案全面评估

  一、整体评价

  方案定位清晰，技术选型务实，MVP 路径合理。但存在几个值得深入思考的维度。

  ---
  二、产品维度评估

  优势

  - 市场定位准确：二次元/互联网亚文化研究是真实需求，目标用户（研究者、运营、内容创作者）明确
  - 数据护城河：历史词频基线是核心竞争力，随时间积累价值越来越高

  待解决的问题

  1. 什么是"爆火"的准确定义？
  - 当前方案用 Current_Freq / Baseline_Freq，但分母为 0 时怎么办？新词本身 Baseline 就是 0
  - 需区分：新词首次出现 vs 旧词二次爆火 vs 季节性词汇（如"春节快乐"每年都高频）

  2. 用户产品形态未定义
  - 是一个 API？一个网站？一个 Bot？一个定期邮件报告？
  - 不同形态决定 Meilisearch 是否够用，或者需要加 FastAPI / Telegram Bot 层

  3. 召回率 vs 精准率的取舍
  - 词频统计容易把"活动关键词"（如某UP主名字突然爆火）误识别为梗
  - 需要更明确的 False Positive 处理策略

  三、架构维度评估

  潜在风险

  ┌───────────────┬────────────────────────────────┬─────────────────────────────────────────────┐
  │     风险      │            当前方案            │                    问题                     │
  ├───────────────┼────────────────────────────────┼─────────────────────────────────────────────┤
  │ B站反爬       │ asyncio.sleep + 代理池（建议） │ 未说明具体策略，这是最大的工程风险          │
  ├───────────────┼────────────────────────────────┼─────────────────────────────────────────────┤
  │ Baseline 存储 │ "临时 Redis 或本地 JSON"       │ 过于模糊，历史数据是核心资产，不能是"临时"  │
  ├───────────────┼────────────────────────────────┼─────────────────────────────────────────────┤
  │ LLM 幻觉      │ 直接信任输出                   │ AI 对生僻梗可能编造来源，需人工审核环节     │
  ├───────────────┼────────────────────────────────┼─────────────────────────────────────────────┤
  │ 数据量级      │ 未评估                         │ B站热门视频评论轻松百万条，分词成本有多大？ │
  └───────────────┴────────────────────────────────┴─────────────────────────────────────────────┘

  架构缺失模块

  当前流程：采集 → 分词 → LLM → Meilisearch
  缺失：    调度器 / 监控 / 去重 / 人工审核 / 数据版本管理

  ---
  四、头脑风暴：其他思路与优化方案

  A. 数据源扩展

  当前：B站评论 + 弹幕
  可加：
    - 微博超话（更快，政治梗/娱乐梗）
    - 抖音评论（音视频梗）
    - NGA论坛（游戏梗）
    - Telegram 频道（小圈子梗，最超前）
    - 小红书（女性向亚文化）
  多源交叉验证可大幅提升置信度：一个词在三个平台同时爆，大概率是真梗。

  B. 更好的"梗发现"算法

  当前方案的局限：词频统计对谐音梗、变体梗、表情包（图片）完全失效。

  替代/补充方案：

  1. 语义聚类替代词频
    - 用 embedding 模型（如 bge-m3）将评论向量化
    - 检测某个语义簇突然密集出现 → 比词频对新词更友好
  2. 异常检测模型
    - 对每个词建立时序模型（Prophet / ARIMA）
    - 检测统计异常点，比简单环比更鲁棒
  3. 引用链追踪
    - 找到"最早使用该词的视频"，自动确定起源
    - B站视频发布时间戳 + 评论时间戳可以做溯源

  C. AI 分析层优化

  当前痛点：单次 LLM 调用，无验证，容易幻觉。

  改进方案：

  # 建议的多步验证流程
  Step 1: 快速筛选 (DeepSeek-V3, batch)
    → 输出: is_meme: bool, confidence: float

  Step 2: 仅对 confidence > 0.7 的词进行深度分析
    → 工作流固定执行火山联网搜索，LLM 只消费搜索结果
    → 输出: MemeRecord with sources[]

  Step 3: 来源交叉验证
    → 检查 sources[] 中的 URL 是否真实存在
    → 这一步可以用简单的 HTTP HEAD 请求完成

  可加入的 AI 能力：
  - 多模态：截取视频帧，识别表情包图片中的文字
  - 关系图谱：梗与梗之间的衍生关系（"父梗" → "子梗"）

  D. 存储与检索层升级

  当前：Meilisearch（全文搜索）

  建议补充：
    - SQLite/PostgreSQL：存储原始词频时序数据（Meilisearch 不适合时序）
    - 向量数据库（Qdrant/Chroma）：支持"找相似的梗"查询
    - 时序数据库（DuckDB）：高效的窗口函数计算词频环比

  数据模型建议增加字段：
  {
    "lifecycle_stage": "emerging | peak | declining | dead",
    "first_detected_at": "2026-03-01",
    "peak_date": "2026-03-10",
    "related_memes": ["依托", "答辩"],
    "source_video_bvid": "BV1xx411c7mD",
    "confidence_score": 0.92,
    "human_verified": false
  }

  E. 产品化方向

  1. Meme 生命周期预测：基于历史数据，预测一个梗还能火多久 → 对内容运营极有价值
  2. 订阅推送：每周 Top 10 新梗邮件/RSS，低成本高价值
  3. API 货币化：对品牌/媒体机构开放查询 API
  4. "梗年鉴"：每年末出一份报告，本身就是传播内容

  ---
  五、优先级建议

  P0（MVP必须）:
    ✓ 确定数据存储方案（不能是"临时JSON"）
    ✓ B站采集的反爬策略
    ✓ Baseline 词频的时间窗口定义

  P1（第一个迭代）:
    - 人工审核界面（哪怕是 CLI）
    - 多步 AI 验证
    - 数据版本管理

  P2（后续规划）:
    - 多平台数据源
    - Embedding 语义检测
    - 梗生命周期模型

  ---
  六、一句话总结

  ▎ 方案的"骨架"已经很好，核心薄弱点在于：Baseline 数据的持久化设计、B站采集的稳定性、以及LLM
  输出的可信度验证——这三个问题不解决，系统会很脆弱。建议在开始编码前先把数据存储层和采集层的方案细化到可执行的程度。
