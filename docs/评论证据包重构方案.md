# MemeDetector 评论证据包重构方案

> 文档状态：草稿
> 更新日期：2026-04-09
> 定位：替代旧版“候选词主轴”的新一代 Miner / Research 设计

---

## 一、背景与问题

当前实现以“候选词”为主轴推进流程：

`评论 -> 候选词提取 -> 是否是梗 -> 生成词条`

这个设计在处理“整句模板梗 + 实体填槽”的评论时容易发生目标错位。典型误判：

- 评论中的真正传播单位是句式模板
- 句尾出现的作品名、角色名、事件名只是填充值
- 旧流程却把显眼的陌生短语抽成候选词，并一路写成最终词条

误例：

- 评论：`闭嘴，如果你惹怒了我，我将会询问有没有推荐的番剧，之后说一代不如一代，并且不讲异国日记！`
- 错误入库：`不讲异国日记`
- 实际问题：`异国日记` 是作品名，真正的梗更可能是前半句的威胁式模板句

本次重构不再兼容旧的“候选词/word”主轴，直接改为“评论证据包 + competing hypotheses + research 裁决”架构。

---

## 二、重构目标

### 目标

1. 让 `miner` 从“评论初筛器”升级为“证据收集器”
2. 让 `research` 从“给定一个词写词条”升级为“比较多个假设并裁决”
3. 显式区分：
- 可复用模板句
- 实体名/作品名/人名
- 填槽项
- 背景语境词
4. 允许 `research` 输出：
- `accept`
- `reject`
- `rewrite_title`
- `manual_review`
- `merge_into_existing`
5. 从架构层面解决“把句式里的作品名误当梗本体”的问题

### 非目标

- 不做旧表结构兼容
- 不保留旧 `candidates` 主轴
- 不保留 `deep_analysis(word, ...)` 这种接口

---

## 三、核心原则

1. `miner` 不做最终定性，只做侦查、拆句、搜索和证据整理
2. `research` 不接受单个词输入，只接受 `bundle / hypothesis bundle`
3. 最终词条标题只能在 `research` 阶段确定
4. 搜索默认不带“梗”等强先验词，先判断“它是什么”，再判断“它是否被梗化”
5. 一个评论里可能同时存在多个 competing hypotheses，必须显式比较，而不是沿着一个 seed 写到底

---

## 四、整体架构

新流程：

`raw video/comments`
-> `insight`
-> `span extraction`
-> `hypothesis construction`
-> `evidence collection`
-> `bundle`
-> `research decision`
-> `record / reject / review`

模块职责：

### Scout

- 拉取视频、评论、视频摘要
- 不做梗判断

### Miner

- 判断评论是否值得侦查
- 从评论中抽取多个 `span`
- 构建一个或多个 `hypothesis`
- 触发选择性搜索
- 汇总视频上下文与网页搜索证据
- 输出 `bundle`

### Research

- 读取 `bundle`
- 比较多个 `hypothesis`
- 决定是否入库、拒绝、改标题、合并或人工复核
- 若通过，生成最终 `record`

---

## 五、核心对象

系统只保留四个业务主对象：

### 1. Insight

含义：
- 一条评论是否值得侦查

特点：
- 是入口，不是词条候选
- 表示“这条评论有信号”，不表示“已经找到梗”

### 2. Span

含义：
- 从评论里切出来的可疑传播片段

作用：
- 区分模板句、引语、填槽项、作品名、人名、背景词

### 3. Hypothesis

含义：
- 对“真正的梗核心是什么”的一个候选判断

作用：
- 作为 `research` 的输入单位
- 支持多个 competing hypotheses 并存

### 4. Decision

含义：
- `research` 对 hypothesis 的最终裁决

作用：
- 决定：
  - 是否入库
  - 是否改标题
  - 是否拒绝
  - 是否并入已有词条
  - 是否转人工复核

### Evidence

说明：
- `Evidence` 不是主对象，而是 `Hypothesis` 的附属证据
- 统一承载视频上下文、网页搜索正证据和反证据

## 七、Miner 输出协议

`miner` 不再输出“候选词列表”，而是输出 `bundle`：

```json
{
  "bundle_id": "bundle_xxx",
  "insight": {},
  "video_refs": [],
  "spans": [],
  "hypotheses": [],
  "hypothesis_spans": [],
  "evidences": [],
  "miner_summary": {
    "recommended_hypothesis_id": "hyp_xxx",
    "should_queue_for_research": true,
    "reason": "评论包含模板句与实体填槽混合结构，已完成基础取证。"
  }
}
```

规则：

1. `bundle` 至少包含一个 `hypothesis`
2. 每个 `hypothesis` 至少绑定一个 `primary` span
3. `recommended_hypothesis_id` 只是建议，不是最终判断
4. `miner` 不能输出 `final_is_meme`、`definition`、`category` 等终局字段

---

## 八、Research 输入输出协议

### 输入

`research` 的输入是单个 `bundle`，或多个相近 bundle 聚合后的 `bundle_group`。

### 输出

```json
{
  "decision": "rewrite_title",
  "target_hypothesis_id": "hyp_xxx",
  "final_title": "闭嘴，如果你惹怒了我……",
  "confidence": 0.84,
  "reason": "传播核心是模板句，而不是句末出现的作品名片段。",
  "record": {}
}
```

约束：

1. `accept` / `rewrite_title` 必须附带最终 `record`
2. `reject` / `manual_review` 可以没有 `record`
3. `final_title` 不要求等于 `candidate_title`
4. `research` 必须允许推翻 `miner` 推荐的 hypothesis

## 九、搜索策略定稿

### 设计原则

- 默认不加 `梗`
- 先查“这个词/片段是什么”
- 再查“它是否被梗化”
- 搜索是为了收集证据，不是为了证明它一定是梗

### 查询顺序

#### 1. `literal`

直接搜原词或原句：

- `异国日记`
- `不讲异国日记`
- `闭嘴 如果你惹怒了我`

#### 2. `contextual`

加中性语境词：

- `异国日记 动漫`
- `异国日记 新番`
- `闭嘴 如果你惹怒了我 表情包`
- `闭嘴 如果你惹怒了我 台词`

#### 3. `origin_probe`

只在前两层不足时，再补：

- `出处`
- `来源`
- `原句`
- `台词`

#### 4. `meme_probe`

最后才补：

- `梗`

说明：
- `meme_probe` 是末级补充，不是默认主查询
- 如果 `literal` / `contextual` 已足以判断对象属性，则停止继续搜索

### 证据权重

高到低：

1. `literal`
2. `contextual`
3. `origin_probe`
4. `meme_probe`

原因：
- 直接搜原词即可命中的事实更可靠
- 必须加 `梗` 才命中的内容更容易被搜索先验带偏

## 十、Prompt 设计要求

### Miner Prompt

核心要求：

1. 不要直接下最终结论
2. 先拆分评论中的多个传播片段
3. 区分：
- 模板句
- 引语
- 填槽项
- 作品名/人名/事件名
- 背景词
4. 搜索是为了收集正反证据
5. 如果片段更像作品名或实体名，要显式写成反证

禁止输出：

- 最终词条定义
- 最终分类
- 最终是否入库

### Research Prompt

核心要求：

1. 不预设任何 hypothesis 就是最终词条
2. 要比较 competing hypotheses
3. 必须允许：
- `reject`
- `rewrite_title`
- `manual_review`
- `merge_into_existing`
4. 最终标题必须对应“真实传播单位”

---

## 十一、裁决规则

### 倾向 `reject`

- 主要证据显示片段是作品名、人名或事件名
- 缺少独立复用证据
- 只在单条评论中成立
- 搜索结果主要是实体介绍，而非传播用法

### 倾向 `rewrite_title`

- 当前 hypothesis 不是梗本体
- 但 bundle 中存在更合理的核心 hypothesis
- 评论中的真正传播单位是模板句或更稳定的表达单元

### 倾向 `manual_review`

- 评论明显像梗，但名称不稳定
- competing hypotheses 证据接近
- 来源太新，搜索证据不足

### 倾向 `merge_into_existing`

- 只是已有梗的局部变体
- 只是某个既有模板下的填槽用法

## 十二、误例回放

评论：

`闭嘴，如果你惹怒了我，我将会询问有没有推荐的番剧，之后说一代不如一代，并且不讲异国日记！`

理想拆解：

- `闭嘴，如果你惹怒了我……` -> `template_core`
- `异国日记` -> `entity_work`
- `不讲异国日记` -> `unknown` 或 `mixed_expression`

理想 hypotheses：

- H1：前半句模板是梗核心
- H2：`不讲异国日记` 本身是梗
- H3：`异国日记` 只是作品填槽项

理想 evidence：

- 对 H1：句式/表情包/台词相关证据
- 对 H2：弱证据或无独立传播证据
- 对 H3：强证据表明其为作品名

理想 decision：

- H1 -> `accept` 或 `rewrite_title`
- H2 -> `reject`
- H3 -> `reject`

## 十三、落地顺序

### 第一阶段：模型与存储重构

1. 删除旧 `candidates` 主轴
2. 新建：
- `comment_insights`
- `comment_spans`
- `hypotheses`
- `hypothesis_spans`
- `evidences`
- `research_decisions`
3. 新建 Pydantic 模型：
- `Insight`
- `Span`
- `Hypothesis`
- `Evidence`
- `MinerBundle`
- `ResearchDecision`

### 第二阶段：Miner 重写

1. 评论初筛
2. span 抽取
3. hypothesis 构建
4. 选择性搜索
5. evidence 汇总
6. 输出 bundle

### 第三阶段：Research 重写

1. 读取 bundle
2. 比较 competing hypotheses
3. 输出 decision
4. 必要时生成 `MemeRecord`

### 第四阶段：测试补齐

重点补回归测试：

- 模板句 + 实体填槽误判
- 热点事件无需加 `梗` 的中性搜索
- `rewrite_title` 路径
- `reject` 与 `manual_review` 路径

---

## 十四、对旧实现的直接结论

以下设计应视为已废弃：

- 以 `word` 为中心的候选词流程
- `CandidateSeed`
- `QuickScreenResult.word -> is_meme`
- `deep_analysis(word, sample_comments, ...)`
- `record.id = word`

新的主轴应为：

`comment -> span -> hypothesis -> evidence -> decision -> record`
